# Booking Form — Complete Conditional Logic Reference

This document is a companion to the main booking form specification. It covers **only the conditional logic** — every "if X then Y", every branch, every state cascade, every validation gate. If something here conflicts with the main spec, this document is authoritative on matters of logic.

This document intentionally excludes visual design, icons, microcopy, and anything other than behavioural rules.

---

## 1. Overview of the form flow

The form has 7 possible steps. **Not every user sees all 7** — the path depends on choices made at Step 2 and Step 5.

```
┌────────────────────────────────────────────────────────────────┐
│                                                                │
│   Step 1       Step 2       Step 3        Step 4               │
│   Unit    →    Role    →    Your     →    AC system            │
│                             details                            │
│                                                                │
│                              ↓                                 │
│                                                                │
│              Step 5: Property access                           │
│              (big conditional tree — see §6)                   │
│                                                                │
│                              ↓                                 │
│                                                                │
│   ┌──────────────────────────┴──────────────────────────┐     │
│   │                                                      │     │
│   │ If access method is one of three "coordination"     │     │
│   │ types → SKIP Step 6                                  │     │
│   │                                                      │     │
│   │ Otherwise → show Step 6 (Schedule)                   │     │
│   │                                                      │     │
│   └──────────────────────────┬──────────────────────────┘     │
│                              ↓                                 │
│                                                                │
│                         Step 7                                 │
│                         Review & pay                           │
│                                                                │
│                              ↓                                 │
│                                                                │
│                       Terminal state                           │
│                   (booking confirmed,                          │
│                  payment cancelled, or                         │
│                   unit unavailable)                            │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

The three "coordination" access types that cause Step 6 to be skipped are listed in §6. For users on those paths, the form has effectively **6 steps, not 7**. The progress indicator should reflect this (e.g. "Step 6 of 6" rather than "Step 7 of 7" on the final step).

## 2. Step 1 — Unit

### 2.1 Unit availability state

Each unit in the dropdown is in one of two states, supplied by the backend:

| State | Behaviour |
|---|---|
| **Available** | User can select it normally. |
| **Already booked** | Visible in the list but **not selectable**. Must carry a visible label indicating it is already booked. |

### 2.2 Validation

- **Required:** one available unit must be selected.
- If the user attempts to advance without a unit selected → error, cannot advance.
- If a unit is somehow in the "already booked" state and the user attempts to select it → must be prevented at the UI level.

### 2.3 Downstream effect

When a unit is selected, it determines:

- Which building the booking is for (affects which schedule slots show up at Step 6 — not user-visible, but architecturally important)
- Which AC pre-loaded values appear at Step 4 (see §5.1)
- Which specific unit address is interpolated into the signature blocks at Step 5 (see §6.5)

---

## 3. Step 2 — Role

### 3.1 The fork

The user picks exactly one of two roles. There is no "Tenant" option (tenants do not book — they are contacted via the coordination flow if applicable).

| Role | Used by |
|---|---|
| **Owner** | The person who owns the unit |
| **Agent / Property Manager** | Someone booking on behalf of an owner they represent |

### 3.2 Validation

- **Required:** one role must be selected.
- No role selection → cannot advance.

### 3.3 Downstream conditionals triggered by this choice

This is the biggest fork in the form. The role selected here changes:

| Downstream element | Owner | Agent / Property Manager |
|---|---|---|
| Agency field on Step 3 | Not shown | **Shown** (required dropdown) |
| Primary-residence sub-question on Step 5 | **Shown** | Not shown (agents never live in the unit) |
| Access method options on Step 5 | Vary by primary-residence answer | One fixed set of 3 options |
| "Pay later" placeholder button on Step 7 | Not shown | **Shown** (disabled, "coming soon") |

### 3.4 Cascade if role is changed after proceeding

If the user goes back to Step 2 and changes role, the following downstream state must be cleared:

- Agency selection (Step 3) — cleared
- Primary residence answer (Step 5) — cleared
- Access method choice (Step 5) — cleared
- All access-method follow-up fields (tenants, signatures, key location, return method, agent dropdown) — cleared
- Service date/slot (Step 6) — cleared

Booker's name, email, phone can stay — they're still the same person.

## 4. Step 3 — Your details

### 4.1 Field visibility by role

| Field | Owner | Agent / Property Manager |
|---|---|---|
| Agency (dropdown from backend agencies list) | — | **Required**, shown first |
| First name | Required | Required |
| Last name | Required | Required |
| Email | Required (valid format) | Required (valid format) |
| Mobile | Required | Required |

When role is Agent, the Agency dropdown appears **before** the name/email/phone fields, not after.

### 4.2 Validation

- All visible required fields must be filled in.
- Email must match a valid email pattern.
- For Agents, the agency must be a non-empty selection from the dropdown.
- If any field fails validation, the error message must indicate which field(s) are invalid.

### 4.3 Cascade if fields are changed

Changes to fields on this step do not affect downstream state (role is unchanged; name/email/phone/agency only affect what's captured, not the flow).

## 5. Step 4 — AC system

### 5.1 Pre-loaded data conditional

When a unit is selected at Step 1, the backend provides that unit's current AC records.

| Unit has records on file? | Behaviour on Step 4 |
|---|---|
| **Yes** (at least one of: number of systems, number of additional indoor units is > 0) | Pre-fill the stepper fields with the stored values. Show a note above the fields confirming this: "Our records from the last service show X systems with Y additional indoor units. Adjust if anything has changed." |
| **No** (all values null or zero) | Defaults apply: 1 system, 0 additional. No pre-load note shown. |

### 5.2 Field behaviour

Both steppers are always shown. The user can always adjust values.

| Field | Min | Max | Default |
|---|---|---|---|
| Number of systems | 1 | 10 | 1 (or pre-loaded) |
| Additional indoor units | 0 | 29 | 0 (or pre-loaded) |

### 5.3 Live price calculation

Total price = (systems × system price) + (additional indoor × add-on price)

System price and add-on price are backend-provided constants. Total updates live as the user adjusts either stepper.

### 5.4 Validation

- Number of systems must be ≥ 1. (Stepper UI prevents going below 1.)
- Additional indoor units must be ≥ 0. (Stepper UI prevents going below 0.)
- No other validation applies.

### 5.5 Cascade if AC values are changed

No downstream state depends on AC values. Changes here only affect the price shown at Step 7.

## 6. Step 5 — Property access (the big one)

This step has three nested layers of conditional logic.

### 6.1 Layer A — Primary residence sub-question (Owner only)

Shown only if role = Owner. Agents skip this sub-question entirely.

| User's answer | Leads to Layer B |
|---|---|
| I live in the property | 3 access options for owner-live-in |
| The property is leased out | 4 access options for owner-leased-out |

### 6.2 Layer B — Access method

The set of access options shown depends on role AND (for owners) primary-residence answer:

#### Owner + "I live in the property" — 3 options

| Access method key | Label |
|---|---|
| `owner_live_at_unit` | I'll be at the unit |
| `owner_live_leave_key` | I'll leave a key with someone |
| `owner_live_collect` | Please collect and return my key |

#### Owner + "The property is leased out" — 4 options

| Access method key | Label |
|---|---|
| `owner_leased_be_there` | I'll be there to provide access |
| `owner_leased_tenant` | Arrange with tenant |
| `owner_leased_agent` | Arrange with agent |
| `owner_leased_leave_key` | I'll leave a key with someone |

#### Agent / Property Manager — 3 options

| Access method key | Label |
|---|---|
| `agent_be_there` | I'll be there to provide access |
| `agent_tenant` | Please arrange with tenant |
| `agent_trade_key` | Please collect & return agent trade key |

### 6.3 Layer C — Follow-up fields per access method

This is the **complete matrix of what each access method requires**. Each row defines one terminal branch — 10 branches total.

| Access method key | Extra fields required | Signature block required? | Informational note shown? |
|---|---|---|---|
| `owner_live_at_unit` | None | No | No |
| `owner_live_leave_key` | Key holder name + key holder phone (both required) | No | Yes — "Leaving a key" note |
| `owner_live_collect` | Key collection location (required free-text), return method (required choice: locker OR hand delivery), signature block | **Yes** — Collect & Return T&Cs | Yes — "Collect & Return Key Service" note |
| `owner_leased_be_there` | None | No | No |
| `owner_leased_tenant` | Tenants list (see §6.4), signature block | **Yes** — Tenant-contact authorisation | Yes — "Tenant details required" note |
| `owner_leased_agent` | Agency (required dropdown from backend agencies list) | No | No |
| `owner_leased_leave_key` | Key holder name + key holder phone (both required) | No | Yes — "Leaving a key" note |
| `agent_be_there` | None | No | No |
| `agent_tenant` | Tenants list (see §6.4), signature block | **Yes** — Tenant-contact authorisation | Yes — "Tenant details required" note |
| `agent_trade_key` | Signature block | **Yes** — Access authorisation | Yes — "Agent trade key" note |

### 6.4 Tenants list behaviour

Used in two access methods: `owner_leased_tenant` and `agent_tenant`.

- Must have at least **one** tenant (adding the first one is mandatory)
- No upper limit on how many tenants can be added
- Each tenant requires: first name, last name, email (valid format), phone — **all four are required per tenant**
- Tenants can be removed individually, but the last remaining tenant cannot be removed (always keep at least one)
- If any tenant has any missing/invalid field, Continue is blocked with an error indicating which tenant is incomplete

### 6.5 Signature blocks

Three distinct signatures exist. Each has the same structural shape but different wording:

| Signature name | Used in access methods |
|---|---|
| Collect & Return T&Cs | `owner_live_collect` |
| Access authorisation | `agent_trade_key` |
| Tenant-contact authorisation | `owner_leased_tenant`, `agent_tenant` |

All three signatures have the same structure:

1. A read-only text block containing the authorisation or T&C wording (may be multi-paragraph; may be long for tenant-contact authorisation which includes cancellation clause — see §10)
2. A required tickbox labelled "I have read and agree to the above"
3. A required typed-name field labelled "Your full name (typed signature)"
4. An auto-filled current date (display-only)

Validation: both the tickbox and the typed name must be present/filled. Missing either blocks Continue.

The signature text may reference the unit address (interpolated from Step 1). This is a content concern handled at the backend wording level.

### 6.6 Additional access notes (always optional, always shown)

A free-text textarea labelled "Additional access notes (optional)" appears regardless of access method. No validation applies — it's always optional.

### 6.7 Validation for Step 5

Step 5 validates conditionally based on the access method chosen. The complete validation rules:

1. If role = Owner: primary residence must be answered. If not, block.
2. Access method must be selected. If not, block.
3. The access method's follow-up fields (per §6.3) must all be valid:
   - Leave-key methods: both name and phone required
   - Collect & return: collection location filled, return method selected, signature valid
   - Tenant methods: at least one tenant with all 4 fields valid, signature valid
   - Arrange-with-agent: agency selected
   - Trade-key: signature valid
   - Be-there methods and at-unit: no extra validation

4. The Additional access notes textarea is never validated (always optional).

### 6.8 Cascade rules within Step 5

Changes to earlier answers on Step 5 cascade to clear later answers:

| If user changes… | Clear… |
|---|---|
| Primary residence (owner sub-question) | Access method + all follow-up fields |
| Access method | All follow-up fields for the previously selected method (tenants list, signature, key holder, return method, agency etc.) |

### 6.9 Cascade to Step 6 (Schedule)

Certain access method choices eliminate Step 6 entirely. See §7.

## 7. Step 6 — Schedule

### 7.1 Skipped or shown?

Step 6 is **conditionally skipped** based on the access method chosen at Step 5.

| Access method | Step 6? |
|---|---|
| `owner_live_at_unit` | Shown |
| `owner_live_leave_key` | Shown |
| `owner_live_collect` | Shown |
| `owner_leased_be_there` | Shown |
| **`owner_leased_tenant`** | **Skipped — coordination flow** |
| **`owner_leased_agent`** | **Skipped — coordination flow** |
| `owner_leased_leave_key` | Shown |
| `agent_be_there` | Shown |
| **`agent_tenant`** | **Skipped — coordination flow** |
| `agent_trade_key` | Shown |

When Step 6 is skipped, the flow goes directly from Step 5 → Step 7. The progress indicator should reflect 6 total steps (not 7) for these flows.

### 7.2 When Step 6 is shown: slot behaviour

- Backend supplies a list of available dates × time windows, with remaining capacity per slot
- Capacity-zero slots are not selectable
- User must select exactly one date + window combination

### 7.3 Validation

- **Required (when step is shown):** a date and window pairing must be selected.

### 7.4 Cascade

If the user goes back to Step 5 and changes access method to/from a coordination type, Step 6 must be shown or hidden accordingly, and any previously selected date/slot cleared if switching into a coordination flow.

## 8. Step 7 — Review & pay

### 8.1 Summary display varies by flow

The review summary lists the same items for all users, but two fields vary:

| Summary item | Scheduled flow (Step 6 was shown) | Coordination flow (Step 6 was skipped) |
|---|---|---|
| Service date | The selected date | **"To be coordinated"** |
| Time window | The selected window | (Not shown) |

Coordination flows also show an explanatory note above or beside the summary: *"We'll contact your tenant(s)/agent after payment to arrange a service time. You'll be emailed with the confirmed date once agreed."*

The date/window pair is replaced by this explanatory message for coordination users.

### 8.2 Cancellation-terms acknowledgement

A visible summary of the cancellation policy is shown, followed by a required tickbox:

- ☐ "I have read and accept the cancellation and rescheduling terms above."

**Until the tickbox is ticked, the Pay button(s) are disabled.**

Applies to all flows (owner + agent, scheduled + coordination). No exceptions.

### 8.3 Payment action varies by role

| Role | Buttons shown |
|---|---|
| Owner | One active button: **Pay now** |
| Agent / Property Manager | Two buttons side by side: **Pay now** (active) + **Pay later (invoice) — coming soon** (visible but disabled) |

The "Pay later" button is purely a placeholder for a future feature. It should be visibly disabled and communicate "coming soon" — clicking it does nothing functional yet.

### 8.4 Submission may fail server-side

Even after the tickbox is ticked and Pay is clicked, the submission can still fail because the chosen unit may have become unavailable in the time between form-start and submit. See §11.

If this happens, the user returns to the form with a visible error near the Pay action:

> "This unit has already been booked. If you believe this is in error, please contact us at hello@taylr.com.au."

The unit is then re-disabled at Step 1.

## 9. Terminal states

After the Pay action, the user lands on one of four states.

| Terminal state | When it occurs | "Book another" action? |
|---|---|---|
| **Booking confirmed — scheduled flow** | Payment succeeded AND step 6 was shown AND date/window chosen | **Yes** |
| **Booking confirmed — coordination flow** | Payment succeeded AND step 6 was skipped | **Yes** |
| **Payment cancelled** | User cancelled on Stripe page | No — offer to retry |
| **Unit unavailable** | Server-side uniqueness check rejected the submission | No — error message only |

### 9.1 "Book another" flow

See §12 for the full rules around carryover.

## 10. Cancellation & rescheduling policy (as it affects the form)

The form does not implement cancellation or rescheduling — those actions happen elsewhere (support email, admin panel, future self-service link). But the form presents the terms at two specific moments:

### 10.1 Step 7 acknowledgement tickbox

Already covered in §8.2. Must be ticked before Pay is enabled.

### 10.2 Tenant-contact authorisation letter content

When the access method is `owner_leased_tenant` or `agent_tenant`, the signed letter text (shown at Step 5 for the booker, and later emailed to tenants) **includes the cancellation clause**. The letter will:

- State the booking is paid and slot is committed
- Reference Residential Tenancies Act access requirement
- Note that rescheduling or cancelling within 48 hours will cost the landlord $125 per unit
- Ask tenants to respond promptly

The form itself doesn't need to know the exact wording — it just needs to accommodate potentially long multi-paragraph signature text. But the logic that the tenant letter **carries the cancellation clause** is captured here so Replit's design accounts for a longer read-only text block in this specific case.

## 11. Uniqueness rules

Conditional rules that prevent duplicate bookings.

### 11.1 Unit uniqueness

| Condition | Effect |
|---|---|
| A unit has a confirmed paid booking for the current service period, not cancelled or refunded | Unit is "already booked" |
| A unit has a coordination-flow booking (paid but unscheduled) | Unit is "already booked" |
| A unit has a pending-unpaid booking attempt | Unit is still available |
| A unit's booking was cancelled or refunded by staff | Unit becomes available again |

### 11.2 Enforcement points

Two checks, both required:

| Check | When | Purpose |
|---|---|---|
| **Soft UI check at Step 1** | When the unit dropdown loads | Prevents most users from starting down a path that can't complete |
| **Hard server-side check at Pay** | Just before launching Stripe | Catches race conditions — two users filling the form simultaneously |

### 11.3 Race condition outcome

If the soft check passed (unit looked available) but the hard check fails (unit got booked by another user in the meantime):

1. The submission is rejected
2. User is returned to the form
3. Error message shown at Step 7
4. The offending unit is re-disabled at Step 1
5. User can select a different unit or exit

## 12. Multi-booking ("Book another") logic

### 12.1 Available from

Both successful terminal states offer a "Book another" action.

### 12.2 What carries over to the new booking

The booker's identity and context — same person, same sitting:

- Role (Owner or Agent)
- Agency (if Agent)
- First name, last name, email, mobile

### 12.3 What resets

Unit-specific data must be fully cleared:

- Unit selection (must be re-chosen)
- AC system counts (will be re-pre-loaded from new unit's records)
- Access method + all follow-up fields
- **Signatures** — critical: signatures are specific to one unit and one service. Must be re-captured even if the same person is signing. No exception.
- Tenants list
- Service date + window
- Cancellation acknowledgement tickbox (must be re-acknowledged for the new booking)

### 12.4 Where the user lands

User lands on Step 1 of a fresh form. Progress indicator resets. Pre-filled fields (role, agency, name, email, phone) are populated behind the scenes, but the user still walks through the flow step by step — they cannot jump directly to Step 5 or similar, because the per-unit considerations (access method, AC, signatures) demand deliberate re-entry.

### 12.5 Uniqueness interaction

The unit just booked in the previous session is now "already booked" and will appear disabled at Step 1 of the new booking. The user cannot book the same unit twice.

## 13. Navigation & cascade summary

### 13.1 Back navigation

- Users can navigate backwards to any already-completed step
- The current step's data is discarded when going back (the user will re-enter it as they come forward again)
- Already-completed steps retain their entered data
- Progress indicator allows clicking any completed (ticked) step to jump directly

### 13.2 Forward navigation

- Forward navigation requires the current step to pass validation
- Forward skipping (jumping ahead without filling intermediate steps) is not allowed
- Exception: the coordination flow automatically jumps from Step 5 to Step 7, skipping Step 6 — this is not user-initiated; it's a flow rule

### 13.3 Complete cascade reset table

When the user changes an answer on an earlier step, later answers that depend on it must be cleared.

| When user changes… | Clear these answers |
|---|---|
| Unit (Step 1) | Pre-loaded AC values at Step 4 (will be re-populated from new unit) |
| Role (Step 2) | Agency (if was Agent), primary residence, access method, access follow-ups, date/slot |
| Agency (Step 3, for Agents) | None downstream |
| Name/email/phone (Step 3) | None |
| AC system values (Step 4) | None |
| Primary residence (Step 5, for Owners) | Access method + all follow-ups |
| Access method (Step 5) | Its follow-up fields (tenants, signature, key holder, return method, agency, etc.) |
| Individual follow-up fields (Step 5) | None (self-contained) |
| Date/slot (Step 6) | None |
| Cancellation tickbox (Step 7) | Disables Pay until re-ticked |

### 13.4 What the cascade does NOT clear

- Booker identity data (role, agency, name, email, phone) is preserved unless the user specifically changes an earlier field that invalidates it
- Anything the user hasn't touched yet on future steps stays blank (not "cleared" because it was never set)

## 14. Validation gates summary

A single-table reference of everything that blocks forward progression.

| Step | Validation gate |
|---|---|
| 1 → 2 | Unit selected, and the selected unit is Available (not Already booked) |
| 2 → 3 | Role selected (Owner or Agent) |
| 3 → 4 | First name, last name, email (valid format), mobile all filled. For Agents, agency also selected. |
| 4 → 5 | None specific (steppers prevent invalid states) |
| 5 → 6 (or 5 → 7 for coordination) | (a) For Owners: primary residence answered. (b) Access method selected. (c) All access-method-specific follow-up fields valid. (d) Signature valid (tickbox + typed name) where required. (e) For tenant flows: at least one tenant with all 4 fields valid. |
| 6 → 7 (when Step 6 is shown) | Date + window pairing selected |
| 7 → Pay action | Cancellation acknowledgement tickbox ticked |
| 7 → Terminal state | Server-side: unit uniqueness check passes; Stripe session creation succeeds |

## 15. State data model (what needs to be persisted during the form session)

For Replit's awareness, the following state must persist across step navigation. This is logical state — storage mechanism (memory, localStorage, session) is Replit's design decision.

| State key | Type | Notes |
|---|---|---|
| `unit_id` | Integer | From Step 1 |
| `role` | Enum | "Owner" or "Agent/Property Manager" |
| `agency_id` | Integer (nullable) | Only set when role = Agent |
| `contact_first_name` | String | Step 3 |
| `contact_last_name` | String | Step 3 |
| `contact_email` | String | Step 3 |
| `contact_phone` | String | Step 3 |
| `num_systems` | Integer 1-10 | Step 4 |
| `num_additional_indoor` | Integer 0-29 | Step 4 |
| `primary_residence` | Enum (nullable) | "live_in" or "leased_out" — only for Owner role |
| `access_method` | Enum | One of the 10 access method keys |
| `key_holder_name` | String (nullable) | Leave-key methods only |
| `key_holder_phone` | String (nullable) | Leave-key methods only |
| `key_collection_location` | String (nullable) | Collect & return only |
| `return_method` | Enum (nullable) | "locker" or "hand_delivery" — collect & return only |
| `managing_agency_id` | Integer (nullable) | `owner_leased_agent` only — the agency to contact |
| `tenants` | Array (nullable) | Tenant methods only — `[{first, last, email, phone}, …]` |
| `signature_acknowledged` | Boolean | Tickbox state on signature block |
| `signature_name` | String (nullable) | Typed name on signature block |
| `access_notes` | String (nullable) | Always-optional textarea |
| `service_date` | String (nullable) | Step 6 — null for coordination flows |
| `service_slot` | String (nullable) | Step 6 — null for coordination flows |
| `cancellation_acknowledged` | Boolean | Step 7 tickbox |

On "Book another" (see §12), all fields are cleared EXCEPT:

- `role`
- `agency_id`
- `contact_first_name`, `contact_last_name`, `contact_email`, `contact_phone`

All others reset to their initial empty state.
